## 1. Claude Codeとは何か

Claude Codeは、Anthropicが開発したターミナルベースのAIコーディングエージェントです。ChatGPTやCopilotとの最大の違いは**エージェント性**にあります。

### 従来のAIコーディングツール（ChatGPT / Copilot）

```
人間 → プロンプト → AI → コード提案 → 人間が実行
```

- 人間が「次に何をするか」を常に指示する
- 一度に一つのタスクしか処理できない
- ファイル操作・コマンド実行は人間が行う

### Claude Code

```
人間 → 目標設定 → Claude Code → [ファイル読込・編集・コマンド実行・検索...] → 成果物
```

- 目標を渡すと、達成するための手順を自分で考えて実行する
- 複数ファイルの同時編集・テスト実行・エラー修正を自律的に行う
- 人間は「何を達成したいか」だけを伝えれば良い

### 実際の違いを体感する例

**ChatGPTへの依頼フロー**:

> 「Expressのルーターを書いて」→コードを貼り付け→「型エラーが出た」→修正案をもらう→貼り付け→「テストが落ちた」→再度質問...（繰り返し、合計15〜30分）

**Claude Codeへの依頼**:

> 「Express + TypeScriptでREST APIを作って。認証はJWT、DBはPostgres。既存のプロジェクト構造に合わせて。テストもVitest で書くこと。」

Claude Codeは既存コードを読み、型定義を確認し、適切なファイルに実装し、テストを書き、エラーがあれば自分で修正します。**所要時間: 5〜10分**。

### なぜ今Claude Codeなのか

2026年現在、AIコーディングツールの競争は激化していますが、Claude Codeが特に優れているのは以下の点です：

- **長いコンテキスト**: 大規模なコードベース全体を理解した上で作業できる
- **ツール使用の正確さ**: ファイル編集・コマンド実行を意図通りに実行できる
- **推論の深さ**: なぜそうするかを考えた上で実装する（バグの根本原因追求など）
- **CLAUDE.mdエコシステム**: プロジェクト固有の文脈を永続的に保持できる

## 2. セットアップと初期設定

### インストール

```bash
npm install -g @anthropic-ai/claude-code
```

Node.js 18以上が必要です。インストール後、バージョンを確認します：

```bash
claude --version
```

### 認証設定

```bash
claude
# 初回起動時にAnthropicアカウントとの認証が走る
# ブラウザが開くのでログインする
```

Anthropicのサブスクリプション（Claude Pro / Max）または API キーが必要です。

### 推奨設定（settings.json）

`~/.claude/settings.json` に以下を設定することで、作業中の確認ダイアログを最小化できます：

```json
{
  "permissions": {
    "allow": [
      "Bash(git:*)",
      "Bash(npm:*)",
      "Bash(npx:*)",
      "Bash(python:*)",
      "Bash(pip:*)",
      "Read",
      "Edit",
      "Write",
      "Glob",
      "Grep"
    ]
  },
  "model": "claude-opus-4-6"
}
```

**ポイント**: 権限を事前に許可しておくことで、作業中断が減り、長いタスクをスムーズに実行できます。本番環境では慎重に設定してください。

### プロジェクトごとの設定

`.claude/settings.local.json`（gitignore推奨）にプロジェクト固有の設定を書けます：

```json
{
  "permissions": {
    "allow": [
      "Bash(make:*)",
      "Bash(docker:*)"
    ]
  }
}
```

## 3. CLAUDE.mdの書き方：AIに「文脈」を渡す技術

Claude Codeが他のAIツールと一線を画す機能が **CLAUDE.md** です。プロジェクトのルートに置くだけで、自動的に読み込まれコンテキストとして機能します。

### 基本構成

```markdown
# プロジェクト名

## 概要
このプロジェクトは○○を目的としたWebアプリです。

## 技術スタック
- Frontend: React 18 + TypeScript
- Backend: Node.js + Express + Prisma
- DB: PostgreSQL 16
- 認証: JWT + Refresh Token

## コーディング規約
- 関数は必ず型アノテーションを付ける
- エラーハンドリングは Result 型を使う
- テストは vitest で書く

## 禁止事項
- `any` 型の使用禁止
- console.log の本番コードへの混入禁止

## よく使うコマンド
- `npm run dev`: 開発サーバー起動
- `npm test`: テスト実行
- `npm run db:migrate`: マイグレーション実行
```

### 上級テクニック1：役割を持たせる

特定のロールを定義することで、より一貫した成果が得られます：

```markdown
## あなたの役割
あなたはシニアバックエンドエンジニアとして振る舞います。
コードレビューでは必ずパフォーマンスと型安全性を優先してください。
パフォーマンスに影響する変更は必ずベンチマーク計測を提案してください。
```

### 上級テクニック2：禁止事項を具体的に書く

```markdown
## 絶対にやってはいけないこと
- 既存のAPIエンドポイントのパスを変更しない
- DB のマイグレーションは自動実行しない（必ず確認を取る）
- package.json のメジャーバージョンを上げない
- src/legacy/ ディレクトリには触れない
```

### 上級テクニック3：よくある作業パターンを記述する

```markdown
## 標準的な作業フロー
新機能追加時:
1. src/types/ に型定義を追加
2. src/services/ にビジネスロジックを実装
3. src/routes/ にエンドポイントを追加
4. src/tests/ にテストを追加
5. docs/api.md にドキュメントを追加
```

### CLAUDE.mdのベストプラクティス

| 項目 | 良い例 | 悪い例 |
|------|--------|--------|
| 禁止事項 | 「src/legacy/を変更しない」 | 「注意して作業する」 |
| 技術スタック | バージョン番号まで記載 | 「Reactを使用」 |
| コマンド | 実際のコマンドを列挙 | 「適切に実行する」 |
| 役割 | 具体的な職種・スタンス | 「エンジニア」 |

## 4. 効果的なプロンプト設計

### 悪いプロンプト vs 良いプロンプト

**悪い例**:
```
ユーザー管理機能を追加して
```

**良い例**:
```
src/routes/users.ts に以下のエンドポイントを追加してください：

- POST /users: ユーザー作成（email, password必須、バリデーション: email形式・パスワード8文字以上）
- GET /users/:id: ユーザー取得（認証必須、自分のデータのみアクセス可）
- PATCH /users/:id: ユーザー更新（name, avatarUrlのみ変更可）
- DELETE /users/:id: ユーザー削除（ソフトデリート）

既存のsrc/middleware/auth.tsの認証ミドルウェアを使うこと。
エラーレスポンスはsrc/utils/error.tsのフォーマットに合わせること。
完了したら src/tests/users.test.ts にテストも追加すること。
```

### プロンプト設計の5原則

1. **具体的なファイルパスを指定する**: 「どこに実装するか」を明示
2. **制約を明示する**: 「既存の○○を使うこと」「○○の変更は不要」
3. **成功基準を示す**: 「○○のテストが通ること」
4. **既存コードを参照させる**: 「src/models/User.tsのパターンに合わせて」
5. **スコープを絞る**: 一度に1〜2個のタスクに集中

### タスクの分割方法

大きなタスクは必ず分割します。例えば「新機能を追加する」は以下のように分けます：

```
✗ 「ECサイトのカート機能を実装して」（1回で全部やらせる）

✓ ステップ1: 「src/types/cart.ts にカート関連の型定義を作って」
✓ ステップ2: 「src/services/cart.ts にカートのビジネスロジックを実装して」
✓ ステップ3: 「src/routes/cart.ts にAPIエンドポイントを追加して」
✓ ステップ4: 「カート機能のテストを書いて」
```

各ステップを確認しながら進めることで、意図しない実装を防げます。

## 5. 実践：実際の開発フローへの組み込み方

### パターン1：バグ修正

```bash
claude "TypeError: Cannot read property 'id' of undefined が src/services/payment.ts:42で発生している。
原因を調査して修正してください。関連するテストも更新すること。
修正前に原因の仮説を述べてから修正に入ること。"
```

**ポイント**: 「修正前に原因の仮説を述べてから」と入れることで、見当違いな修正を防げます。

### パターン2：コードレビュー

```bash
claude "今日のコミット(git diff main)をレビューして。
パフォーマンス問題・型安全性・セキュリティの観点で問題があれば指摘して、
重大な問題は自動で修正して。軽微な問題はコメントのみ。"
```

### パターン3：リファクタリング

```bash
claude "src/utils/helpers.ts が肥大化している。
関数を適切なモジュールに分割して、既存のimportを壊さないように移動させて。
変更前に分割方針を提案し、承認してから実行すること。"
```

### パターン4：ドキュメント生成

```bash
claude "src/api/ 配下のすべてのエンドポイントを読んで、
OpenAPI 3.0 仕様書を docs/api.yaml に生成して。
既存のapi.yamlがあれば差分だけ更新すること。"
```

### パターン5：テスト自動生成

```bash
claude "src/services/order.ts のユニットテストをvitest で書いて。
テストカバレッジ80%以上を目標に。
エッジケース（空のカート・在庫切れ・決済失敗）を必ず含めること。"
```

### 日常的なワークフローへの組み込み

以下のように**毎日のルーティン**に組み込むと効果的です：

```bash
# 朝: 前日のコードをレビュー
claude "git diff HEAD~1 のコードをレビューして、問題があれば修正して"

# 実装中: 完成度チェック
claude "現在の実装でTODOコメントが残っているファイルをリストアップして、対処法を提案して"

# 退勤前: テスト通過確認
claude "npm testを実行して、失敗したテストがあれば修正して"
```

## 6. サブエージェント活用：並列処理で開発速度を加速

Claude Codeの最も強力な機能の一つが **サブエージェント**（Agent ツール）です。

### 基本的な使い方

複数の独立したタスクを並列で実行できます：

```
メインClaude ─┬─→ サブエージェントA（フロントエンド実装）
              ├─→ サブエージェントB（バックエンドAPI実装）
              └─→ サブエージェントC（テスト作成）
```

これにより、順次実行なら30分かかるタスクが10分で完了することもあります。

### 実装例：CLAUDE.mdへのサブエージェント指示

```markdown
## サブエージェント方針
- フロントエンドとバックエンドの実装は必ず並列で行う
- テスト作成は実装と並列で実施する
- 独立したモジュールの実装は並列化を優先する
- 並列タスクは必ず完了を確認してからマージする
```

### サブエージェントが効果的なシーン

| シーン | 効果 |
|--------|------|
| フロント/バック分離開発 | 両方同時に実装 |
| 複数ページのドキュメント生成 | 並列で一括生成 |
| マルチサービス対応のリファクタリング | 独立して並列処理 |
| テスト・実装の同時進行 | TDDサイクルを高速化 |

## 7. 応用パターン：チーム開発・CI/CDへの展開

### チーム開発への導入

チームにClaude Codeを導入する際のベストプラクティス：

**共有CLAUDE.mdの管理**

```markdown
# チーム共有 CLAUDE.md

## コーディング規約（チーム合意済み）
- ...

## アーキテクチャの原則
- レイヤードアーキテクチャを維持する
- ドメインロジックはservices/に、インフラはinfra/に置く

## レビュー基準
- PRは必ずテストを含む
- パフォーマンスに影響する変更はベンチマーク結果を添付
```

**個人設定の分離**

```
プロジェクトルート/
  CLAUDE.md          ← チーム共有（git管理）
  .claude/
    settings.local.json  ← 個人設定（gitignore）
```

### CI/CDパイプラインとの統合

GitHub Actionsでのコードレビュー自動化の例：

```yaml
name: AI Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code
      - name: Run Review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude --print "git diff origin/main...HEAD のコードをレビューして。
          セキュリティ問題・バグ・パフォーマンス問題があればGitHub Issueとして報告できる形式でまとめて。"
```

### 定型作業の自動化スクリプト

```bash
#!/bin/bash
# review.sh - 毎日の自動レビュースクリプト

echo "=== 日次コードレビュー開始 ==="
claude --print "
本日のコミット（git log --since='24 hours ago' --oneline）を確認して以下を実施：
1. 追加されたコードのレビュー（セキュリティ・型安全性）
2. テストカバレッジの確認
3. TODO/FIXMEコメントのリストアップ
結果をレポート形式で出力して。
"
```

### 大規模コードベースでの活用

大きなリポジトリでは、作業スコープを明示することが重要です：

```bash
# 特定ディレクトリに限定
claude "src/features/auth/ 配下のコードのみを対象に、
型エラーをすべて修正して。他のディレクトリは変更しないこと。"

# 特定ファイル種別に限定
claude "*.test.ts ファイルのみを確認して、
テストカバレッジが低い（50%未満の）ファイルをリストアップして"
```

## 8. よくあるミスと回避法

### ミス1：コンテキスト切れ

**症状**: 長いタスクの途中でClaudeが「前の作業を覚えていない」状態になる

**対策**:
- `/compact` コマンドで会話を要約する
- 重要な情報はCLAUDE.mdに書いておく
- 大きなタスクは最初から複数のステップに分割する

```bash
# 長い作業前の準備
claude "作業開始前に現状を把握するため、
プロジェクト構造・主要ファイル・依存関係を調査してサマリーをCLAUDE.mdに追記して。"
```

### ミス2：許可のたびに中断される

**症状**: ファイル編集のたびに確認ダイアログが出て作業が止まる

**対策**:
```bash
# セッション中のみ全許可
claude --dangerously-skip-permissions

# または settings.json で事前に許可リストを設定（推奨）
```

settings.jsonの許可リスト設定が最も安全で効果的です。

### ミス3：意図しないファイルへの変更

**症状**: 関係ないファイルまで変更される

**対策**:
```
「src/components/Button.tsx のみを変更すること。他のファイルは触らないこと。
src/styles/ ディレクトリは絶対に変更禁止。」
```

スコープを明示する習慣が重要です。また、CLAUDE.mdの禁止事項セクションに常設で書くのも有効です。

### ミス4：ループへの陥り

**症状**: 同じエラーを修正→再発を繰り返す

**対策**:
```
「このエラーが3回繰り返されています。根本原因を調査してから修正方針を提案してください。
すぐに修正しないこと。原因が特定できない場合はその旨を報告して。」
```

### ミス5：過度に依存しすぎる

**症状**: Claude Codeが書いたコードを理解せずにマージしてしまう

**対策**:
- 重要な変更は必ず説明を求める
- 「変更内容を箇条書きで説明してから実装して」と指示する
- 本番環境へのデプロイ前に必ず自分でコードレビューする

```bash
claude "実装を開始する前に、変更するファイルと変更内容の概要を
マークダウン形式で説明してください。確認後に実装に進みます。"
```

### ミス6：エラーメッセージなしで依頼する

**症状**: 「動かない」だけ伝えてもClaude Codeが的外れな修正をする

**対策**:
```
✗ 「ログインが動かない」

✓ 「ログインボタンを押すと以下のエラーが出ます：
   POST /api/auth/login 401 Unauthorized
   {"error": "invalid_credentials"}
   ネットワークタブでは正しいJSONが送信されていることを確認済み。
   src/services/auth.ts のtokenValidation関数が怪しいと思う。調査して。」
```

## 9. 実践例：実際のプロジェクトで起きた変化

### 事例1：バグ修正時間の劇的削減

**before**: 本番障害のデバッグに平均3時間
**after**: Claude Codeにログとスタックトレースを渡すと平均40分で解決

具体的なプロンプト:
```
以下のエラーログを分析して原因を特定してください：

[ERROR] 2026-04-15 14:23:11 Payment processing failed
  at src/services/payment.ts:128:15
  TypeError: Cannot read properties of undefined (reading 'customerId')
  
関連する最近のコミット（git log -10）も確認して原因を推測してください。
```

### 事例2：テスト整備の加速

**before**: テストカバレッジ23%、新機能にテストを書く時間がない
**after**: 1週間でカバレッジ78%に（既存コードにテストを追加）

プロセス:
```bash
# ステップ1: カバレッジの低いファイルを特定
claude "npm run coverage の結果を分析して、
カバレッジが50%未満のファイルをリストアップして優先度順に並べて"

# ステップ2: 一つずつテスト追加
claude "src/services/order.ts のテストを vitest で書いて。
エッジケース（空注文・在庫切れ・支払い失敗）を必ず含めること"
```

### 事例3：レガシーコードのドキュメント化

**before**: 5年前のコードを誰も理解できない、ドキュメントなし
**after**: 2日でコンポーネント図・API仕様書・処理フロー説明書が完成

```bash
claude "src/legacy/ ディレクトリのコードを読んで、
以下のドキュメントを docs/legacy/ に作成してください：
1. コンポーネント間の依存関係図（Mermaid形式）
2. 各ファイルの責務の説明（2〜3行）
3. 変更する際の注意点リスト"
```

### 事例4：新人エンジニアのオンボーディング改善

CLAUDE.mdを活用した新人向けのガイド機能：

```markdown
## 新人エンジニア向けガイド
新しいメンバーがこのプロジェクトに参加した場合:
- まず README.md を読むよう案内する
- よくある質問は FAQ.md を参照する
- わからないことがあれば具体的に教えてあげる姿勢で
```

新人が「このコードは何をしているの？」と聞くと、プロジェクトの文脈を知った上で適切に説明してくれます。

## 10. 次のステップ：自律的なAI組織を構築する

Claude Codeの可能性は単一エージェントの活用にとどまりません。**複数のClaude Codeインスタンスを組織的に連携**させることで、ソフトウェア開発チーム全体を自動化できます。

### マルチエージェント構成例

```
agenta（受付・振り分け）
    ├── manager（タスク管理・優先度付け）
    │       └── PM（プロジェクト管理）
    │               ├── dev（コーディング）
    │               ├── qa（テスト・品質管理）
    │               └── docs（ドキュメント）
    └── review（コードレビュー）
```

各役割に専用のCLAUDE.mdを持たせ、成果物をファイル経由でやり取りすることで、完全自律的な開発パイプラインが実現します。

### 実装のポイント

1. **役割の明確化**: 各エージェントのCLAUDE.mdに責務・禁止事項を明記
2. **成果物の規格化**: ファイル名・保存場所・フォーマットを統一
3. **エスカレーション設計**: 判断不能な場合の上位エージェントへの伝達経路
4. **オーナーへの通知**: Telegram等での重要事項の人間への報告

### AI組織構築の第一歩

まずは小さく始めます：

```bash
# Step 1: 単一エージェントの自律実行を試す
claude --print "以下のタスクを自律的に実行して報告書を作成して：
1. テストを実行（npm test）
2. カバレッジが50%未満のファイルをリストアップ
3. 最もカバレッジが低い3ファイルにテストを追加
4. 改善前後のカバレッジを report.md に記録"

# Step 2: cronで定期実行
# 0 9 * * 1 claude --print "週次コードレビュー..." >> logs/weekly_review.md
```

### 今日から始めるアクションリスト

1. **今すぐ**: プロジェクトのルートに `CLAUDE.md` を作成する
2. **今日中**: 最も時間がかかっているルーティンタスクをClaude Codeに委譲してみる
3. **今週中**: `settings.json` で権限設定を最適化し、フルオートで動かす
4. **今月中**: マルチエージェント構成の設計図を書いてみる
