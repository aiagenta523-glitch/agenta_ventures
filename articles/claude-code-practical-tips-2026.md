## 設定篇：まずここを整えると劇的に変わる

### 1. CLAUDE.md は「コンテキスト圧縮ファイル」として使う

CLAUDE.mdを「指示書」として使っている人が多いですが、私が一番効果を感じたのは「プロジェクトのコンテキストを詰め込むファイル」として使うことです。

```markdown
# プロジェクト概要
- 目的：エンジニア向けキャリア相談AIサービス
- 技術スタック：Python 3.11 / Streamlit / Anthropic API
- ユーザー：転職検討中のエンジニア（25〜40歳）

# 設計思想
- シンプルさ優先：過度な抽象化より、読みやすいコード
- コスト意識：Anthropic APIのコストはプロンプトキャッシュで最小化

# 直近のコンテキスト
- 現在：Gumroadライセンス認証機能を実装中
- 完了：MVP実装・Streamlit Cloudデプロイ
- 次：テスト追加
```

「このプロジェクトでは何を、なぜ作っているか」を書いておくことで、Claude Codeが毎回適切な判断をしてくれます。

### 2. `/init` コマンドは新しいセッションのたびに使う

長いセッションを続けるより、コンテキストが詰まってきたら `/init` でリセットする方が精度が上がります。

CLAUDE.mdに十分な情報があれば、リセット後も即座に適切な提案をしてくれます。

### 3. `--max-tokens` より「段階的なタスク分割」が効く

長い実装を一気にやらせるより、こまめに区切った方が精度・速度ともに上です：

```
× 「ユーザー認証機能全体を実装して」
○ 「まず認証フローの設計をリストアップして」→確認→「最初のステップだけ実装して」
```

## プロンプト篇：言い方ひとつで結果が変わる

### 4. 「〜してください」より「〜という問題を解決したい」

```
× 「このコードをリファクタリングしてください」

○ 「このコードは読みにくく、チームメンバーが理解するのに時間がかかっています。
   特に、ライセンス検証ロジックが分散していることが問題です。
   どう改善するか提案してください」
```

問題の背景を共有すると、的外れな修正を避けられます。

### 5. 「制約条件」を先に伝える

```
以下の制約でAPIクライアントを実装してください：
- Python 3.9以上で動作すること
- 依存ライブラリは最小限（標準ライブラリ + requests のみ）
- リトライ処理とタイムアウト設定を含める
- 既存の `config.py` の設定を使うこと

実装してください。
```

制約を後から言うと、やり直しが発生します。

### 6. 「説明してから実装する」フローが最も精度が高い

複雑な実装の場合：

```
実装する前に、どういうアプローチを取るか説明してください。
コードは書かずに、方針とトレードオフだけ。
```

→ 方針を確認・修正する

```
OK、その方針で実装してください。
```

このフローだと、設計ミスのまま長いコードを書かれることを防げます。

### 7. コードレビューの依頼は「観点を指定する」

```
このコードをレビューしてください。
特に以下の観点で問題があれば指摘してください：
1. セキュリティ（インジェクション、認証バイパス等）
2. エラーハンドリングの漏れ
3. パフォーマンスボトルネック

良い点を挙げる必要はありません。問題点だけ教えてください。
```

「全体的にレビューして」より、指摘の精度と数が上がります。

## ワークフロー篇：毎日の開発サイクルに組み込む

### 8. 朝一のコンテキスト復元は「タスク一覧から」

```
昨日の作業内容: [コミットログを貼り付け]

今日やること:
1. Gumroadライセンス検証のユニットテスト追加
2. Streamlitのセッション管理バグ修正
3. デプロイ設定の更新

まず1から始めます。どこから手をつけると良いか提案してください。
```

昨日のコンテキストを渡すことで、ゼロから説明する必要がなくなります。

### 9. バグ調査は「再現ステップ + 期待値 + 実際の挙動」の3点セット

```
バグを調査してください。

再現ステップ:
1. Streamlitアプリを起動
2. メールアドレスを入力してログイン
3. 「キャリア相談を始める」をクリック

期待される挙動: チャット画面に遷移する
実際の挙動: 「ライセンスキーが無効です」エラーが出る（初回ログイン時のみ）

関連するコード: [license.py の verify_license 関数を貼る]
```

情報が揃っていると、Claude Codeは原因候補を絞り込んで提案してくれます。

### 10. 「やってほしいこと」と「やってほしくないこと」を同時に伝える

```
app.py のエラーハンドリングを改善してください。

やってほしいこと：
- try/except が足りない箇所の追加
- エラーメッセージをユーザー向けに分かりやすく変更

やってほしくないこと：
- 関数の分割・リファクタリング（別タスクで扱う）
- ログ出力の変更（loggingの設定は触らない）
```

「やってほしくないこと」を書かないと、余計な変更が入ることがあります。

## 失敗事例：やってみて効果がなかった・逆効果だったこと

### ✗ 「賢く使って」「最善の方法で」

曖昧な指示ほど、結果がブレます。具体的に書いた方が良いです。

### ✗ 一度に複数の大きなタスクを並べる

```
× 「認証機能・API連携・フロントエンド修正・テスト追加をまとめてやってください」
```

コンテキストが混在して、どれも中途半端になります。

### ✗ エラーメッセージだけを貼って「直して」

```
× 「エラーが出ました → [エラーログ貼り付け]」
```

周辺コードと「何をしようとしていたか」が分からないと、的外れな修正になります。

## まとめ：Claude Codeは「道具」、使い方で結果が10倍変わる

Claude Codeは確かに優秀ですが、使い方次第で結果の差が大きい道具です。

効果が高かった習慣まとめ：
- **CLAUDE.md にコンテキストを詰める**
- **制約条件を先に伝える**
- **実装前に方針を確認する**
- **やってほしくないことも明示する**
- **バグ報告は3点セット（再現・期待・実際）**

プロンプトを丁寧に書く手間は、やり直しのコスト削減に直結します。
